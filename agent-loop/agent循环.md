## 从一个贪吃蛇游戏开始演进


### FAQ
#### Q1：传给LLM接口的messages为什么需要规范化？什么是规范化？
- 如何规范化？
  + 定义统一的消息格式（内部规范），为每个LLM提供商写一个转换函数：internal_messages -> provider_specific_messages
  + 使用开源SDK，比如：LangChain，LlamaIndex，LiteLLM

#### Q2：多Agent的几种方案
  + Task工具方式。像定义tool工具一样，可以在调用llm接口时，定义task参数
  + 手动编排方式。像游戏生成agent一样，一个agent负责生成游戏说明书，一个负责根据说明书生成代码，一个负责根据说明书和代码生成测试用例，在这种业务场景中，有明确的流水线，适合使用手动编排的方式
  + 其他的还有对等协商/辩论式，动态委派/层级式，并行投票/集成式，动态自组织/涌现式等等
  
两种方案对比
| 特性 | Task 工具方式 | 手动编排方式 |
|------|--------------|------------|
| **灵活性** | ⭐⭐⭐⭐⭐ 父 Agent 动态决策调用哪个子 Agent | ⭐⭐⭐ 固定流程 |
| **可控性** | ⭐⭐⭐ Agent 自主决策，结果可能不一致 | ⭐⭐⭐⭐⭐ 完全控制执行顺序 |
| **调试难度** | ⭐⭐ 难以追踪 Agent 决策过程 | ⭐⭐⭐⭐ 清晰的调用栈 |
| **Token 消耗** | ⭐⭐ 父 Agent 也需要消耗 token 做决策 | ⭐⭐⭐⭐ 更少 token |
| **适用场景** | 复杂、动态的任务路由 | 明确的流水线任务 |
